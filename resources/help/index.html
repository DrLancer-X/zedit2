Introduction

zedit2 is an editor for the 1991 DOS GCS "ZZT" and the 1992 DOS GCS "Super ZZT". While these game creation systems come with their own editors, zedit2 is designed to be a tool for power users, presenting the ability to access non-standard features that may be useful to designers (e.g. stat stacking, off-board stat placement), as well as productivity features (e.g. mouse drawing, board atlassing, image conversion, buffers, gradient fill.)

It is possible (in fact, fairly easy) to create worlds that will crash or freeze the original ZZT with this tool, and while some behaviours come with a warning, relatively few actions are prohibited. zedit2 is designed to be a tool that can create practically any possible ZZT world, reducing the requirement for manual hex editing to gain access to advanced functionality.

This help file will describe many of the available pieces of functionality in zedit2. Note that, when the help file refers to specific keystrokes, it is referring to the default keystroke configuration. If you have reconfigured any of the keystrokes, references to those will be inaccurate.

Editor overview

  1. Window title - shows the current world being edited and the current board
  2. Menu - provides access to various editing features
  3. Info box - Presents the following information, in order:
    • The number of stats on the board, excluding the player
    • The X and Y position of the cursor
    • The current board's index number and name
    • The size of the current board, in bytes (above 20000 is risky in vanilla ZZT and Super ZZT)
    • The size of the entire world, in bytes (above 430000 may be risky in vanilla ZZT and Super ZZT)
  4. Under cursor tile info - shows information (colour and type) about the tile that is currently underneath the cursor. If there are stats at this location, the box will also show the following stat information for each stat:
    • Stat ID: The index of this stat. 0 is usually the player.
    • Cycle: The cycle this stat runs at (a stat at cycle X will usually execute every X frames.)
    • X-Step: The x-step of this stat.
    • Y-Step: The y-step of this stat.
    • Param 1: The first parameter of this stat.
    • Param 2: The second parameter of this stat.
    • Param 3: The third parameter of this stat.
    • Follower: This stat's follower (used by centipedes).
    • Leader: This stat's leader (used by centipedes).
    • Instr. Ptr: This current position of the code that this stat is executing. -1 means stopped.
    • Code length: The length, in bytes, of this stat's ZZT-OOP code.
    • Under: The colour and type of the tile under this stat.
  5. Buffer tile info - as before, but for the tile currently in the editing buffer. Pressing space will place this tile.
  6. Current mode display - shows the current editing mode:
    • Editing: The default mode.
    • Type to place text: Toggled by F2, when printing characters are typed into the editor, text tiles will be written.
    • Drawing: Toggled by tab, if the cursor is moved with the arrow keys while in this mode, the buffer tile will be placed automatically with each step.
    • ENTER on other corner: Enabled with Alt-B, when block selection mode is enabled, move the cursor to specify the bounds of the block, and press ENTER to access block actions.
    • ENTER to place block: After copying or moving a block, press ENTER to place it at the cursor position.
    Messages will also appear here briefly after performing certain actions such as saving, taking screenshots etc.
  7. Editing screen - the screen where the current board (or boards, if atlassing is enabled) is shown. The bevelled box in the centre is the cursor. If the area is larger than the editing window size, the scroll bars can be used to scroll the display around.

Upon starting up Zedit2, you will be presented with a brand new ZZT world. You can create a new Super ZZT world or open an existing world through the File menu.

The arrow keys will move the editing cursor around the screen. To move the cursor around in steps of 10, you can hold the Alt key down while moving the cursor.

On the left panel in a box titled "Buffer" you will see the current tile you have in your buffer. To place this tile, press space. To choose a different tile, press F3 through F10 to open up different menus containing different tiles that can be placed. Some tiles will require some configuration before you can place them. Tigers, for example, will require you to select an intelligence and firing rate, as well as whether the tiger fires bullets or stars. To choose a different colour, press C.

Press enter when the cursor is over a tile to reconfigure it and store it in your buffer. You can also simply store the tile by pressing insert. If the tile is already in your buffer and you wish to reconfigure it, press P. If you instead want to reconfigure a tile that is on the board without storing it, press Alt-M.

You can perform more advanced editing of a particular tile, including freely setting all stat variables, adding stats to a tile that is not normally meant to have them, removing stats from a tile that is normally meant to have them and adding multiple stats to a single tile by holding Ctrl when you press enter, P or Alt-M. See the advanced editor section of the help file for more details.

You can also use the mouse for editing. Hold the left mouse button and draw to place tiles. Use the right mouse button to store the tile under your mouse position into your buffer. Both of these will also move your cursor to the mouse position. You can also move the cursor to the mouse position without placing or storing a tile by pressing the middle mouse button. Finally, you can use the mouse wheel to zoom in and out.

File menu actions

New ZZT / Super ZZT World

Creates a new ZZT (or Super ZZT) world. This new world will be created with two boards- a board titled 'Title Screen' and a board titled 'First board', with the player approximately placed in the centre and surrounded by empties.

Open world

Allows you to choose a .ZZT, .SZT or .SAV world to open. Whether the file is ZZT or Super ZZT is discovered by analysing the file header.

Recent files

Shows the 10 most recent files opened in Zedit2 for quick access.

One world at a time

If this option is selected Zedit2 will only be able to open one world at a time and loading or creating a new world will close the current one. If this option is not selected you can open up multiple worlds at a time.

Save / Save as

Saves the current world to the specified location. If the world has previously been opened or saved, 'Save' will automatically resave the file to the same location. If 'Save as' has been selected or the world is a newly created one, a file save dialog will be presented allowing you to choose a location and filename.

When saving, Zedit2 will check your world against various limits and may warn you about settings that may cause compatibility problems when run in (Super) ZZT. Some limits are hard and Zedit2 will be unable to save your world at all if you go past them.

Zedit2 settings

Allows the configuration of several aspects of Zedit2. See the Zedit2 settings section for more information.

Close world

This will simply close the current world being edited.

World menu actions

World settings

Here you can configure various options that are global to the world being edited:

  • Starting board: This is the board the ZZT world will start on when run. In the original ZZT editor the starting board is set to whichever board was being edited when the world was saved. In Zedit2, however, the board needs to be manually selected here.
  • World name: This is the 'world name' of this ZZT world. Note that, unless you change this, it will be automatically set to the filename you save this world as, minus the file extension. This is used internally by ZZT to reload the world after exiting it. Be careful when modifying this.
  • Health (and other counters): Sets the starting value for these counters.
  • Savegame / Locked: In ZZT save files have this 'savegame' byte set automatically. This prevents editing by the original editor unless debug mode is on.
  • Keys: Select which keys the player starts of with here.
  • Flags: Enter the starting flags (up to 10 in ZZT and 16 in Super ZZT). Note that only flags containing only the characters A-Z, 0-9, : and _ can be reference in ZZT-OOP; however, you can set other flags here.

Import world

Opens a file chooser allowing you to select another (Super) ZZT world. All the boards from that world will then be appended onto the world you are currently editing.

Load charset

Allows you to open either a MegaZeux-format .CHR or FontMania-format .COM file to replace the current character set. This character set will also be used when testing worlds if the appropriate configuration options are set.

Note that the loaded character set is not stored in the world format anywhere- if you are working on a world with a custom character set, for now you will need to reload it each time you open Zedit2.

Load palette

Allows you to open a MegaZeux-format VGA .PAL palette file to replace the current palette. This palette will also be used when testing worlds if the appropriate configuration options are set.

Note that the loaded palette is not stored in the world format anywhere- if you are working on a world with a custom palette, for now you will need to reload it each time you open Zedit2.

Default charset and palette

Resets the current charset and palette back to the ZZT defaults.

Blinking

When this is set (the default) Zedit2 will treat colours 128-255 as blinking colours, which is how ZZT has looked for the majority of players. Disabling this will cause Zedit2 to treat the blink bit as a high colour bit instead, resulting in a set of brighter colours being made available instead. This will be used when testing worlds if the appropriate configuration options are set.

Test world

This will save a temporary copy of the world and attempt to launch ZZT to play it. You will need to configure Zedit2 for your environment under 'Zedit2 Settings' in the 'File' menu before you will be able to use this.

Atlas

This toggles board atlassing, which can be a powerful tool for visualising a set of interconnected boards, especially in conjunction with the zoom controls. When you enable atlassing, Zedit2 will start from the board you are currently editing and expand outwards, connecting to boards via the board exits, then display them all in one continuous editing view. Note that boards are not necessarily reciprocally connected, and boards are not always connected in a way that makes cartographic sense. As a result, board atlassing may have different results depending on which board you started from if the boards are not sensibly connected.

Board atlassing causes many editing features to slightly change in terms of behaviour while it is enabled. For example, if you move the cursor into the 'void' between boards and then create a new board, that board will be automatically added to the atlas in the location your cursor was. Furthermore, if that location shared borders with existing boards (and those boards were not connected to anything else in the direction of the newly-added board) those boards will automatically be connected together. This can speed up the creation of large maps.

After an atlas has been created, it will stick around in your current editing session until removed. If you change to another board that is not within an atlas, the editing view will switch back to the single board view; however, upon changing back to a board that is part of the atlas, the atlas view will come back. In addition, you can have multiple separate atlasses created at the same time, if the world contains multiple separate contiguous regions of boards.

To remove an atlas, simply switch to a board that is part of an atlas, then choose Atlas from the World menu again (or press Ctrl-A).

Note that large altasses may have a serious impact on editing performance. If this is causing problems you might find it better to stick with the single-board view while editing your world and only use atlas mode for looking at your work afterwards.

Board menu actions

Board settings

Modifies various per-board settings:

  • Board name: The name of the current board. This is never shown in-game, but is used in ZZT editors to help with keeping track of boards.
  • Max player shots: Maximum number of player bullets that can be on the board at a time. If this is set to 0, the player cannot shoot at all. If this is set to another value, the player will not be able to shoot if there are at least this many player bullets on the board.
  • Time limit: The board time limit. When the time limit runs out, the player will lose 10 health. Note that the remaining time is not displayed in Super ZZT. It still works, though.
  • Player entry X/Y: The player's re-entry location, used to keep track of the player's location for the purposes of the 'restart if hurt' flag. Can normally be safely left at 0,0.
  • Camera X/Y: The current location of the player's viewport camera in Super ZZT.
  • Dark: Whether the board is dark or not. This option is not available in Super ZZT.
  • Restart if hurt: If this flag is set and the player takes built-in damage (that is, damage from a bullet, star, creature, blinkwall or from running out of time), the player will be sent back to the location they entered the board on.

Board exits

Allows you to select the boards that can be reached from the various board edges. The 'Reciprocated' check boxes indicate which of these connections are reciprocated from the connecting board (that is, if board A has an east connection to board B and board B has a west connection to board A, they are considered to be reciprocating.) You can also check this box to automatically cause the linked board to link the current board back.

Switch board

Brings up a list of boards, allowing you to switch to editing another board.

Add board

Prompts for a board name, then creates a new board using that name and switches to it.

Add boards (*x* grid)

Prompts for a width, height and board name template, then creates width*height boards, all interconnected in a grid organisation. This can be used for quickly constructing large world map areas and locations. The board name template will have the {x} replaced with the board's X coordinate within this grid (from 1 to width) and the {y} replaced with the board's Y coordinate (from 1 to height).

Two additional options are available. The 'Erase player from boards' checkbox will, if selected, pre-remove the player (as though the Erase player from board operation was performed on each board). The 'Open board grid in atlas view' will open the newly-constructed grid in an atlas.

Board list

This dialog shows a list of all the boards in your currently edited world. From here you can edit the board name, player shots, time limit, player re-entry coordinates, camera coordinates, exits, darkness and restart if hurt settings en masse.

One or more boards can also be selected in this dialog. While boards are selected, the buttons on the right respectively can be used to move the board(s) up or down in the board listing, or delete them. This will automatically edit any board exits or passages pointing to the affected board(s). If destructive changes are made (virtually guaranteed to happen when deleting a board, but can also happen if rearranging boards causes the title screen board to change) the editor will warn you of these changes before prompting you to continue.

Note that passages without a stat (which always lead to the title screen board) will not be altered, and the same is true for any stats that become passages later, so this method of rearranging boards is not entirely foolproof.

Import board

Prompts you to choose a .BRD file to import, loading it and its settings to replace the current board. Note that you will not be able to load a Super ZZT .BRD in a ZZT world and vice-versa.

Export board

Prompts you to choose a .BRD file to export the current board and its settings to. Note that this will not perform any compatibility checks.

Stats list

This dialog shows a list of all the stats and their parameters on the current board. Unlike the board list this does not allow you to directly edit the parameters; however, you can use the columns to sort the data in order to find things quickly.

You can modify a stat by double-clicking on its entry, or right-clicking the entry and selecting 'Modify', or selecting the entry and pressing enter. If you want to access the advanced modification screen, select 'Modify (advanced)' or press ctrl-enter instead.

Any stats you select on this screen will also have a blue rectangle drawn around them in the board view, so you can use this to find where particular stats are. Another thing this stat display allows you to do is find stats that are positioned off-board and modify them, something that cannot be done from the main editor.

Edit menu actions

Select colour

This dialog will let you select a colour to change your current buffer tile to. You can use the arrow keys and enter to select one, or you can use the mouse to click on a colour. Note that, if your current buffer tile is a text element, you will only have a limited palette of colours to choose from. If you wish to change the text character, press P or select 'Modify buffer tile' instead.

Modify buffer tile

If your current buffer tile has associated configurable properties, this will pop up a dialog prompting you to edit it. The nature of this dialog depends on the type of tile selected:

  • Creatures and other stat elements with basic configurable properties will give you a dialog to modify these settings, which are usually things like intelligence, firing rate, direction etc.
  • Objects will prompt you to select the character to represent them, and then will open up their ZZT-OOP code for editing. More information about this dialog is found in the code editor section.
  • Scrolls will open up the Code Editor allowing you to edit their contents.
  • Passages will prompt you to edit their destinations.
  • Text elements will prompt you to select their character.
  • Other types without configurable properties will not result in any dialog being opened.
  • Finally, any tile with more than one property will cause the advanced editor to be opened.

The '(advanced)' form of this option will open up the advanced editor for that tile.

Modify tile under cursor

Similar to 'Modify buffer tile', but this time, the tile under the cursor is modified instead.

Start block operation

Start block operation, more conveniently accessed via the keyboard shortcut (Alt-B), begins the selection of a contiguous region of tiles. Upon selecting the block and pressing Enter, a popup will prompt you to select one of several block actions:

  • Copy block: Allows you to select a location to paste a copy of this block. Everything within the block (tiles, stats) will be duplicated. If a non-clone player is in the block, that player will be moved to the new location. Note that 'Copy block' can also be used to transfer regions between worlds- just select 'Copy block' in the source world, then switch over to the destination world to paste it.
  • Copy block (repeated): Same as 'Copy block', except you can paste the block multiple times. Once you are done, press escape to end the operation.
  • Move block: This will move a region of the board to another location, leaving empty tiles in its wake. Note that, providing the move takes place entirely within the one board, the stat order of those stats being moved will remain unchanged. Move block cannot be used to move data between worlds, however.
  • Clear block: This will erase the contents of the selected region, erasing all stats (except the player) and replacing all tiles with empties.
  • Flip block: This will perform a vertical 'flip' of the contents of the block, such that tiles that were previously at the top are now at the bottom and vice-versa. Like move, this will not change the order of stats providing the entire flip remains within the same board.
  • Mirror block: Similar to 'Flip block', but horizontal.
  • Paint block: This will change the colour of every tile in the block to the colour of the buffer tile. Note that text elements may only be able to change to an approximation of that colour if the colour is not a valid text colour.

While it is not one of the options listed on the menu, you can also press ctrl-(0-9) to store the currently selected block in an extended buffer. See the extended buffers section for more detail. You can also do this after choosing 'Copy block' but before pasting the block.

Enter text

While in this mode (which can also be accessed by pressing F2), characters typed into the editing window will no longer perform their usual keyboard shortcuts, but will instead create text elements of the character typed. Pressing enter while in this mode will also move the cursor down 1 tile vertically, and return the horizontal position to whatever it was when you started typing, allowing you to type in paragraphs naturally. Press escape to leave this mode.

Note that, if your current buffer tile was set to an appropriate text colour (any colour with a white foreground except for white on grey), that colour will be used for text. If it is set to an invalid colour, an approximation will be used instead.

Toggle drawing

While in this mode (which can also be accessed by pressing Tab), moving the cursor around will also place tiles, as though you were pressing space after each movement. Press escape to leave this mode.

Flood fill

Flood fill, also accessible by pressing F, will take the current buffer tile and attempt to place it to cover all identical tiles in a contiguous region. Note that, for the purposes of flood fill, all Empty tiles are considered to be identical irrespective of colour, including empties with stats, so be warned.

Gradient fill

Similar to flood fill, but opens up a dialog allowing you to perform a special kind of flood. See the gradient fill section of this help file for more details.

Convert image

Prompts you to choose a BMP, GIF, PNG or JPEG image to convert into ZZT. See the convert image section of the help file for details.

Convert image from clipboard

As before, but will use whatever image is currently in your system clipboard as input, rather than prompting you to choose a file on your file system.

Erase player from board

This operation will replace the current player on the board with whatever tile was under that player and then move the player stat to the corner between the top and right board edges (61,0 in ZZT, 97,0 in Super ZZT). This is useful for title screens, and occasionally for working on a board's graphical assets without the player getting in the way. If the board is going to be used for gameplay, however, the player should be readded to it first. This can be done simply by creating a new player (in the F10 menu by default) - note that a player clone will not suffice.

Buffer manager

Opens up the extended buffer manager. See the extended buffers section of the help file for more information about this.

Element menus (F3-F10)

These menus, also available via the associated function key, allow you to select various elements to replace your current buffer tile with. These menus are configurable via the Zedit2 settings dialog.

View menu actions

Zoom in / out / Reset zoom

These options will zoom the display in, out and back to the default (1:1) zoom level respectively, allowing you to get an eagle eye view of a large area (useful with Super ZZT or board atlassing), or alternatively get close up for fine editing. You can also scroll the mouse wheel to change the zoom level.

Show grid in atlas

When this is selected, in atlas mode, grid lines will be shown between board boundaries. These can be turned off to enable a more immersive viewing/editing experience. These grid lines, however, do show some useful functional information when it comes to board connectivity:

  • Thin partially translucent green edges mean that this board is connected to the board it is physically next to in the atlas.
  • Thick partially translucent red edges mean this board is connected to some other board via this edge. This board may appear somewhere else in the atlas, for example.
  • Thick opaque white edges mean this board is not connected to anything via this edge.

Show _______

These options, also available via their key combinations (shift-F1 through F5), visually highlight the presence of certain element, which may be difficult to see or tell apart from other elements normally:

  • Show stats: This will visually highlight tiles with an associated stat
  • Show objects: This will visually highlight all Object elements
  • Show invisibles: This will reveal invisible walls
  • Show empties: This will visually highlight all empties
  • Show fakes: This will visually highlight all fakes
  • Show empties as text: This will display empties as text characters, using their colour to determine the text character to display. This can be useful for revealing erased text characters in ZZT games made with the original editor.
  • Show nothing: This will return the display to normal. You can also return the display to normal if currently in a Show _______ mode by selecting that same mode again (either via menu or key combo) or by pressing escape.

Take screenshot

This will take a screenshot of the current board (or boards, if atlassing is on) and save it as a PNG image somewhere on your file system (usually the directory you launched Zedit2 from). This will not include any of the UI or any other visual markup applied by Zedit2, and will always be captured at a 1:1 zoom level.

Take screenshot to clipboard

As before, but this will capture the same shot directly into your system clipboard, for pasting directly into image editors or other software.

Help menu actions

Help

Displays this help document.

About

Shows the current version of Zedit2 in an animated display. Press any key or click on it to close it.

Zedit2 settings

This dialog, available under the 'File' menu, allows you to redefine the functionality of several parts of Zedit2.

Keystroke configuration

Most of the key combinations used in Zedit2 can be configured here. Simply click on one of the boxes and press the desired key combination to change it to that. Note that you will not be able to select a key combination that is already in use elsewhere.

World testing configuration

You can test the world you are editing in Zedit2 by pressing Alt-T or choosing 'Test world' from the World menu. This screen allows you to configure ZZT and Super ZZT for testing purposes.

Note that much of the functionality was designed with asie's Zeta emulator in mind, and while you may have some success with other emulators like Dosbox, or even ZZT game engine recreations, some functionality may be missing.

  • Start the tested world on the current board - The temporarily world created during testing will have its starting board set to the current board if this option is selected. This can make testing a world while working on it more efficient.
  • Test directory: For best results, this should be a directory that:
    1. Can be written to by Zedit2
    2. Contains ZZT and your ZZT emulation solution of choice
    A temporary testing world will be saved into this directory and then your emulator will be invoked to run ZZT with this testing world. The world will be deleted after exiting ZZT (however, if Zedit2 closes for some other reason before ZZT exits, the testing world may be left behind.)
  • Command to run: This is the program that Zedit2 will invoke to test ZZT. It will be invoked from the test directory specified above, so it must be reachable from within that directory.
  • Command parameters: These are command-line parameters that will be passed to the above program (before Zedit2 passes in other parameters, such as the filename of the testing world.)
  • Base name: The name that the temporary testing world will be named after. Note that Zedit2 will attempt to append numbers onto the end of this filename if the file already exists. If the base name is __TEST, the testing world will be named __TEST.ZZT (or __TEST.SZT in the case of Super ZZT). If there is already a file named __TEST.ZZT it will be named __TEST2.ZZT instead, and so on.
  • After ____ milliseconds inject 'P' into the input stream: This is a useful, albeit experimental option designed to be used alongside the '-t' command-line argument for Zeta which will skip the K C [enter] prompt at the start of ZZT. With this, Zedit2 will attempt to press P on the Zeta window when it pops up, launching you directly into the game.
  • After ____ milliseconds inject 'Enter' into the input stream: This is another experimental option designed to be used with the above option when testing Super ZZT worlds. Super ZZT normally launches an introductory sequence after the user presses P, which is then terminated by pressing Enter. This will press Enter for you, allowing you to immediately jump into your Super ZZT world. (Note that the time delay will happen after P is pressed by the above option, assuming it is selected.
  • Use charset and palette, if modified: If a non-default charset or palette is currenty loaded, Zedit2 will save the charset and/or palette to files in the test directory and pass in -l palette:pal:__TEST.PAL and -l charset:chr:__TEST.CHR command-line parameters (with the filename changed to whatever the configured base name is). This will work in Zeta but is unlikely to work in other emulators.
  • If blinking is disabled, pass -b: If Blinking has been turned off (the option is found in the World menu), this will pass the -b command-line option, which in Zeta will disable blinking and treat the blink bit as a high colour bit instead. Again, unlikely to work in other emulators.

Code editor configuration

This screen allows you to configure the colours used in the ZZT-OOP editor. You can also configure whether newline characters are automatically added to the end of objects you edit (as omitting these can occasionally result in unexpected behaviour.)

Element menu configuration

Here you can configure the contents of the F3-F10 menus by dragging elements from one menu to another and renaming the menus as you please. You can also choose some presets inspired by the original (Super) ZZT editor and KevEdit.

Note that both elements from ZZT and Super ZZT will appear on this configuration screen, but they will only appear in the menus if supported by the current world being edited.

The advanced editor

The advanced editor allows you to make nonstandard modifications to a tile's contents. It is divided into two panels; the 'Edit tile' panel and the 'Edit stats' panel. The 'Edit tile' panel simply allows you to change the type and colour of this tile. You can even access types that have no meaning in ZZT (these may cause your game to crash when displayed.)

The 'Edit stats' panel will only be displayed if this tile has stats. You can add a stat to a statless tile by clicking the 'Add stat' button. From here you can select which stat you are editing on the list on the left (in the case of multiple stats associated with one tile) and edit the stat parameters on the right. Most of these are straightforward, but there are a couple of things that stand out:

  • Location: This will change the location, on the board, of this stat. After you apply this change to the board, the stat will move from it. Note that this will change the stat's location, not the tile's. You can make use of this to do things like place stats in off-board locations.
  • Order: The order in which stats execute is important in ZZT, and this is a special stat parameter that is only used in Zedit2, which influences the order of stats. Stats with a lower Order value will be sorted before stats with a higher Order. Note that other editors, including ZZT itself, will erase this setting. This setting also has a different behaviour if the 'Set ID' flag is enabled. Note that stat 0 (the player) will not be reordered irrespective of the value of this parameter.
  • Follower / Leader: You can click the magnifying glass to select a stat from the stat list.
  • Instr. Ptr: The code position in the title bar of the code editor can be used as a reference for editing this value. You can also set it to -1 to 'pre-stop' an object, allowing you to omit a starting #end command.
  • Edit code: This will allow you to edit the code attached to this stat. Notably, this will even allow you to add code to stats that do not normally have any use for it.
  • Bind: This will prompt you to select a stat from the stat list. The stat will then be bound to that stat's code. This is a powerful tool for saving memory, but can be risky when it comes to zap/restore.
  • Autobind: This enables 'autobind' mode for this stat. A stat with autobind will automatically bind with another stat that contains identical code. This can make it easier to quickly bind many identical stats.
  • Set ID: When this flag is enabled, the Order value is treated as an absolute stat number (useful in certain situations, like when dealing with duplicator teleportation or non-1 cycle values.) If the stat number cannot be obtained, Zedit2 will attempt to give the stat a number close to it.
  • Player: The stat 0 player has special treatment in ZZT. This flag identifies this stat as being stat 0.

The code editor

This editor is launched whenever you attempt to edit the code attached to a stat. It contains some basic editing features that you would expect in a text editor, in addition to the ability to preview ZZT #play statements - place the text cursor at the start of the #play line or lines and press Alt-T.

The editor will attempt to warn you about syntactically incorrect ZZT-OOP or otherwise risky practices by displaying text in a warning colour. In that event, you can hold the mouse over the relevant piece of code to see a description of the issue.

There are a couple of other areas where holding the mouse over a section of code will display something useful:

  • Holding the mouse over a #char statement will reveal the character corresponding to the value used.
  • Holding the mouse over a #play statement will reveal the number of /is that a Cycle 1 object will need to wait, on ZZT's default speed, to equal the amount of time that #play sequence will take.

Note that you can press F3 while in the code editor to insert a text character into your code. You might also find this dialog useful for finding the character # of any character, for use with the #char command.

Extended buffers

In addition to the standard editor buffer, which contains one tile and does not persist if Zedit2 is closed, there are a set of extended buffers that can contain individual tiles and blocks of tiles. As long as Zedit2 is closed normally, these buffers will persist along with Zedit2's configuration and can therefore be used to build up a library of useful objects, patterns, fades etc.

You can press Ctrl-(0-9) to store your current buffer tile in one of ten extended buffers. Once you have a tile stored in this way, you can retrieve it by pressing 0-9. It will then become your new current buffer tile.

The extended buffer can be used for storing blocks of tiles as well. While selecting or pasting a block, you can press Ctrl-(0-9). This will store the block in that buffer. When you press 0-9, if there is a block larger than 1x1 in that extended buffer, the editor will switch into block paste mode and you will be able to place it by pressing enter.

You can preview, access and manipulate the contents of your extended buffers by bringing up the buffer manager (Ctrl-B or under the 'Edit' menu). You can then see the contents of each buffer, as well as move them around by clicking and dragging, and retrieve them (as though you pressed the key associated with that buffer) by clicking on them.

Although the first 10 extended buffers are special as they can be accessed through keystrokes, additional extended buffers are available from the buffer manager. You can set their contents by copying something to one of the first 10 buffers using the appropriate key combo, then moving it to another slot in the buffer manager.

Note that buffers with certain properties will be available as patterns for the gradient fill command.

Gradient fill

This feature works similarly to flood fill in how it determines the region to cover, but can be configured to fill that region in special ways. This is a productivity tool designed to help create certain effects that are often seen in ZZT games but are time-consuming to create by hand.

The basic principle here is that you choose a gradient and gradient style and configure it how you please. The flood-filled region will then be filled in using this gradient. The editing window will update as you change the fill settings, so you can preview what the change will look like before you make it.

Gradient styles

  • Linear: A linear gradient drawn along the angle of your choice.
  • Box: Each tile in a box gradient will be drawn based on its Manhatten distance from the defined centre point.
  • Radial: Same as a box gradient, but with Euclidean distance, resulting in a circular gradient.
  • Conical: Tiles are drawn based on their angle away from the centre point instead of distance.
  • Slime: Tiles are drawn based on their distance from the edges of the flood-fill. Useful for creating pools, cave walls etc.

Gradient

Here you can select the gradient that will be used for this fill. Seven default gradients are provided, but if these are not sufficient, you can create additional gradients by storing a block that is more than 1 tile wide and exactly 1 tile high into an extended buffer. All blocks meeting this requirement will be available for selection from the gradient drop-down box.

Gradient parameters

  • Start: How far along in the gradient pattern does the gradient being drawn actually start from. This is normalised such that 0 is the start of the gradient pattern and 1 is the end. If you set this to a value larger than 0, the start of the gradient will be skipped. If you set this to a value smaller than 0, the start of the gradient will effectively be repeated, based on how far into the negatives this goes.
  • Finish: How far along the pattern does the gradient go before ending. If you set this to less than 1, the gradient will end early and the latter parts of the pattern will be unused. If you set this to larger than 1, the last tile in the pattern will be repeated, dependng on how high you set it. Note that, if the 'Start' value is lower than the 'Finish' value, 'Finish' will be treated as 'Start' and vice-versa. If you want to reverse the pattern, use the Invert setting.
  • Horizontal / Vertical: The centre point, relative to the total area being flood-filled. This position is used by the Box, Radial and Conical gradients.
  • Angle: For linear gradients, the angle that this gradient is drawn alongside. For conical gradients, this angle is used to offset the angle the gradient is drawn at.
  • Repeats: This setting causes the gradient pattern to repeat the specified number of times. Note that, on each odd-numbered repeat, the gradient pattern will be reversed.
  • X/Y-Weight: Used by the Slime gradient, this affects how much it "costs" for the fill to travel horizontally vs vertically. The default settings of 0.8 and 1.4 are designed to counteract the rectangular shape of the ZZT tiles. If you are working in Super ZZT you may prefer weights of 1.6 and 1.4, to reflect the wider tiles.
  • Diffusion: Adds a small amount of random variance to the gradient, making it 'noisy'. This is a setting that should probably be fine-tuned after you have chosen all your other flood-fill parameters, but can create visually powerful results.
  • Invert: If checked, the gradient pattern is treated as though it were reversed.

Convert image

This is another productivity tool, designed to take external bitmap images and convert them into a rough visual approximation in ZZT, using ZZT elements as graphical tiles.

Note that this feature is highly experimental and may require significant tuning to produce decent results, if it is possible to produce decent results at all. It may be best to use this as a base for doing manual design work on.

You can access this tool from the 'Convert image' and 'Convert image from clipboard' options in the 'Edit' menu. After specifying an image, it will be loaded into the left screen in the convert image dialog. A preview of the current conversion will be displayed in the right screen.

The converter is affected by the currently loaded palette and charset, as well as the current setting of the 'Blink' parameter. These will all be used in the optimisation process, so it is recommended that, if you are using a custom charset/palette/blink setting, that you have them loaded before you convert your images. Colours are compared in CIELAB colour space with an assumption that the palette is in sRGB colour space, so this assumption may also affect the outcome of the conversion.

Left, right, top, bottom

These values are used to specify the bounds of the source image. You can tweak these to clip the image prior to conversion.

Macroblock width and height

These are important values when it comes to influencing the quality of the conversion, but their use is subtle. These values are the size of the blocks used for comparing the results of the conversion to the original image for the purposes of optimising output.

8x14, for example, will result in entire characters being compared at a time. What will happen in that case is that, in the output image, the contents of an entire tile will be averaged down to one pixel. The respective area in the input image will also be averaged down to a single pixel and the two pixels will be compared for similarity. The element and colour combination that result in the closest pixel will then be chosen to represent that block of graphical data.

If you choose a smaller macroblock size, however, this will result in each char turning into multiple pixels, which will then be compared against pixels created from the same-sized macroblocks in the original image. This can go all the way down to 1x1 macroblocks, which result in individual pixels being compared. The converter will compare the RMSE of different tile choices to choose the most optimal tile to represent that area of the image.

Macroblock sizes can only be provided in factors of the character size (8x14). This means the width of a macroblock can only be 1, 2, 4 or 8 and the height can only be 1, 2, 7 or 14. The macroblock size setting has the following implications:

  • Larger macroblocks result in faster conversion due to fewer comparisons needing to be made.
  • The largest macroblock size (8x14) usually only results in visually pleasing results when only solid and dither characters are in use. This is because the only information used by the converter is the % of pixels in the character that are set. Allowing the use of other elements results in something that looks like it came out of AAlib, which is probably not desirable unless you want that particular effect.
  • The smallest macroblock size (1x1) also frequently results in less-than-ideal results, owing to the fact that individual comparison of pixels does not manage any kind of visual approximation, which means that dither characters, for example, are highly unlikely to be used despite the fact that they are applicable in a lot of cases. However, 1x1 macroblocks do have one particular edge case in which they are useful: If you have a perfect (that is, no resizing, filtering or compression artifacts) screenshot of a ZZT board, you should be able to perfectly recreate it using a 1x1 macroblock size, as this will favour exact matches for chars over anything else. As long as your selected elements and 'Max Stats' value are sufficient for representing the characters used, you can exactly recreate the screenshot in ZZT.
  • It is recommended that you try different settings. Often 2x2 is a reasonable compromise between precision and accuracy.

Output width and height

This is the width and height, in chars, of the output image. Note that the input image will be scaled, if necessary, to match this width and height.

Element use

Here you can select which ZZT elements the converter will use. Generally speaking, the more options you give it, the better it will be able to approximate your image- however, if you want to avoid certain elements due to how you plan on using the result, or because those elements visually result in something you do not like, you can turn those off.

At the bottom of the selection are a number of buttons for preset element configurations. 'Select all' and 'Deselect all' are self-explanatory. 'Dither only' will select only Empty, Water (Floor in Super ZZT), Solid, Normal and Breakable. 'All gfx' will select all of the graphically useful tile types, avoiding a few tiles that have issues (e.g. they change colours or chars during display, or they aren't handled properly by the converter).

Note that all the elements used here are statless, and they will therefore not contribute towards the board stat limit. If you want to make use of stats, that is handled separately, though the 'Max Stats' parameter.

The performance of the converter will be affected by the number of elements you have selected (unless the 'Max Stats' parameter is set to something other than 0), as the more elements the optimiser has to choose from, the more work it will need to do.

Some elements are not recommended, or some with some caveats:

  • Transporter, Object: Without stats these elements are invisible, so there is no point in using them.
  • Star, Clockwise, Counter, SpinningGun: These elements use inconsistent characters (and in the case of star, inconsistent colours) which will change as parts of the screen get refreshed, and are typically not suitable for ZZT art.
  • Line, Web: These elements are not handled properly by the converter- the converter, which only operates on one tile at a time, is unable to optimise based on tiles that are affected by surrounding tiles as these are.
  • Scroll, blinking text (and possibly others): These will cause ZZT to crash or behave oddly if touched, so they should only be used if well out of reach of the player. Other elements may also have undesirable effects if touched as well.

Max stats

This setting allows you to set the maximum amount of stat objects that will be used to allow the converter to better approximate certain characters. Stats count towards the board total and will also result in increased memory use compared to regular elements, so it is recommended that these be used sparingly.

The converter will always use objects for this setting, irrespective of the 'element use' settings. This is because objects are the most optimal way of representing any char/colour pair with a single stat.

If 'Max Stats' is set to anything other than 0, conversion performance will be seriously reduced. This is due to the way the conversion process works. Rather than simply finding the best element/colour combination of every tile, the converter will also find the best char/colour combination. After all of the tiles have been converted to one of the selected elements, the converter will then choose the tile that would result in the greatest reduction of RMSE if it were replaced with an object. The converter will continue replacing the tiles that would result in the most improvement until either it exhausts the 'Max Stats' value or none of the remaining tiles can be improved. As a result, there is a negligible performance penalty from increasing the 'Max Stats' past 1- most of the performance is lost from the initial global optimisation. It is therefore suggested that you tune the other parameters with 'Max stats' set to 0, and when you are happy with that, put in the maximum number of stats you are happy to expend.

Other notes

The image converter is multithreaded and will benefit from additional CPU cores, due to the optimisation happening character at a time. This does, however, mean that the converter is unable to consider how characters can affect the visual perception of neighbouring characters (the principle behind error diffusion and many other quantisation approaches), which means there are bounds on how good the results can be. Note that you can interrupt the conversion by changing settings at any time- you do not need to wait for the conversion to complete.

After the conversion is done and you are happy with it, clicking the 'OK' button will result in the convert image dialog disappearing and you will be able to place the newly-converted image on the board by pressing enter. You can also store the image in an extended buffer by pressing Ctrl-(0-9).

Other miscellaneous info

  • Holding shift while pressing a directional arrow key will switch boards to the board linked in that direction.
  • Holding ctrl-shift while pressing a directional arrow will do the same thing, unless there is no board linked in that direction. In that event, the 'Add new board' dialog will pop up, and the newly created board will automatically be reciprocally linked to the board you were on before in that direction.
  • Home and end keys can be used to quickly jump to the top-left and bottom-right of the board. This also works in the colour and character select screens.
  • When in the character selection screen (for selecting an object or text element's char), you can type in a character to jump to it immediately.

Future work

This is a small wish list for features to go into future versions of Zedit2. Some of these are small things that would be easy to implement but are low priority. Others are 'nice to have's that may have significant issues getting in the way of their implementation.

  • Support for loading and saving of MZM files. This is a file format from MegaZeux that is suitable for holding blocks of tiles.
  • Ability to configure the screenshot save location and filename. These are already configurable, but the configuration screen doesn't have settings for them yet.
  • Some form of support for editing layers. This is a feature Dr Dos has mentioned as a useful feature before- it's just a matter of scoping exactly what is required here.
  • Some ability to configure the limits shown for various warnings. With the Reconstruction of ZZT, modified versions and source ports that do not have the same limits as traditional ZZT may become relevant.
  • Some ability to store the loaded palette/charset/blink setting in the ZZT file itself so these do not need to be loaded each time.
  • A feature that will take screenshots of every board in a world automatically and store them in a directory.
  • Some advanced world querying tools, perhaps similar to those used in the Museum (e.g. being able to search every object's code would be very useful for development.
  • Some advanced world-modifying tools. A global object size optimiser (rewrite labels and flags to be shorter, remove unnecessary line breaks and comments, replace #end with \0 etc.) would be nice, for example.
  • Some kind of external plugin or scripting language support. There are a lot of things that are just much easier to do programmatically when designing a ZZT world and this may be one way of supporting these.
  • Being able to flood-fill 2D patterns would be nice.
  • Support for some kind of non-standard file format...
    • ...that would support world that can't be stored in the ZZT file format (e.g. objects with 100kb of code, more than 255 boards etc.)
    • ...that are human-readable or easier to create tools for, e.g. JSON, XML.
  • Support for a full-screen editing mode that is better than simply maximising the window.
  • Ability to import/export files form the code editor. Currently you can copy and paste the text between programs, but as Zedit2 works in CP437-converted-to-Unicode compatibility may not be the best.
  • Use the currently loaded character set in the code editor and other places where the CP437 font is used. It seems that only vector fonts are supported by Swing so there may not be any easy way of doing this, but it would be nice.
  • Improve the image converter to better consider the global effect of tiles on the visual perception of the image. At the moment it essentially does a local optimisation per tile.
  • Improve performance, especially of general editing while in atlas mode (especially when zoomed out so large portions of the screen are visible at once.)
  • There are some hugely inefficient things Zedit2 does, such as resorting all stats whenever anything relating to a stat is changed, but these have mostly been ignored as they do not appear to have a large contribution to Zedit2's performance problems.
  • Better custom keystroke handling. At the moment only one key combo can be associated with each command and each command can only have one key combo associated with it. This should basically all be handled in a more intelligent way.
  • The config file should have its entries sorted so that the file is human-editable.
  • This help file should have search support.
